The Big Picture

What problem are we solving and why does it matter?

What is a Payment Gateway?

Imagine you're buying a course online. You click "Pay" -- but how does the money move from YOUR wallet to the SELLER's account?

You (Buyer)  -->  ???  -->  Seller (Merchant)

That "???" is the payment gateway. It's a middleman that:

  1. Securely collects your payment info (wallet number, PIN)
  2. Talks to your bank/wallet provider to deduct money
  3. Transfers money to the seller's account
  4. Tells the seller "payment is done, deliver the product"

Real-world Analogy

Think of a cashier at a store:

  • You give money to the cashier (not directly to the product manufacturer)
  • The cashier verifies the money is real
  • The cashier gives you a receipt
  • The store gets the money later through settlement

bKash payment gateway = digital cashier

Tech Stack

Why we chose these tools and what each one does.

Node.js -- The Runtime ▶

What: JavaScript running outside the browser, on the server.

Why:

  • Same language (JS) for frontend and backend
  • Non-blocking I/O -- great for handling many API calls (like calling bKash)
  • Huge ecosystem (npm packages for everything)

In our project: Node.js runs our Express server.

Express.js -- The Web Framework ▶

What: A minimal framework for building HTTP APIs in Node.js.

Core concept -- Routes:

// When someone sends GET request to /api/products,
// run this function and send back product data
app.get('/api/products', (req, res) => {
    const products = getProductsFromDB();
    res.json(products);  // Send JSON response
});

Core concept -- Middleware:

// Middleware = functions that run BEFORE your route handler
app.use(express.json());  // Parses JSON request bodies
app.use(cors());           // Allows cross-origin requests

// Request flow:
// Client -> cors() -> json() -> YOUR ROUTE HANDLER -> Response
SQLite (via better-sqlite3) -- The Database ▶

What: A file-based SQL database. No server needed.

Why: Zero setup, data in a single file, uses standard SQL.

In production, you'd use PostgreSQL or MySQL.

better-sqlite3 (what we use)
const product = db.prepare(
  'SELECT * FROM products WHERE id = ?'
).get(1);
sqlite3 (callback mess)
db.get('SELECT * ...', [1],
  (err, row) => {
    if (err) { /* handle */ }
    // use row here
});
Axios -- HTTP Client ▶

What: A library for making HTTP requests from Node.js (or browser).

Why: Cleaner than built-in http, auto JSON parsing, timeout support (critical for payments).

const { data } = await axios.post(
    'https://bkash-sandbox-url/tokenized/checkout/create',
    { amount: '499', currency: 'BDT' },       // Request body
    { headers: { Authorization: token } }       // Headers
);
// `data` automatically contains the parsed JSON response
dotenv -- Environment Variables ▶

Loads variables from a .env file into process.env. Keeps secrets OUT of your code.

# .env file (NEVER commit this)
BKASH_APP_KEY=4f6o0cjiki2rfm34kfdadl1eqq
BKASH_APP_SECRET=2is7hdktrekvrbljjh44ll3d9l1dtjo4pasmjvs5vl5qr3fug4b

# In your code:
const key = process.env.BKASH_APP_KEY;
CORS -- Cross-Origin Resource Sharing ▶

Problem: React runs on localhost:5173, API on localhost:5000. Browsers block cross-origin requests by default.

Solution:

app.use(cors({ origin: 'http://localhost:5173' }));

Without this, every API call from the frontend would fail.

Why separate orders and payments? ▶
  1. An order can exist WITHOUT a payment (user abandoned checkout)
  2. A failed payment might be retried -- multiple payment attempts for one order
  3. Separation of concerns -- order = WHAT was bought, payment = HOW it was paid

This is called normalization in database design.

Why store price in order_items AND products? ▶

Because product prices can change! If a course costs 499 BDT today and 599 BDT tomorrow, the order should remember the price at the time of purchase.

This is called capturing the price at transaction time.

Payment Gateway -- Core Concepts

Tokenized checkout, tokens, and the three-step payment dance.

What is "Tokenized Checkout"?

  1. We call bKash API --> get a payment URL
  2. We redirect user to that URL (bKash's own page)
  3. User enters their credentials on bKash's page
  4. bKash redirects back to us

Why "tokenized"? Because we never see the user's real credentials. We work with tokens:

TokenPurpose
id_tokenProves WE are a legitimate merchant
paymentIDIdentifies a specific payment transaction
trxIDFinal transaction ID after payment completes

Why tokens instead of password on every call?

Bad: No tokens

Every API call: "Hey bKash, I'm merchant X, password Y, do Z"

Password sent 100 times = 100 chances for interception

Good: With tokens

Login once: get a token. Every call: use the token.

Password sent once. Token is temporary (1 hour).

The Three-Step Payment Dance

Step 1: CREATE

Merchant: "I want to charge customer 499 BDT"

bKash: "OK, here's a payment page URL. Send the customer there."

Step 2: AUTHORIZE

Customer: enters PIN + OTP on bKash's page

bKash: "Customer approved! Redirecting them back to you."

Step 3: EXECUTE

Merchant: "Great, finalize the payment"

bKash: "Done! Here's the transaction ID: TRX12345"

Why not just one step?

Security and user consent. CREATE doesn't move money -- it reserves intent. User must AUTHORIZE on bKash's secure page. Then EXECUTE actually transfers. This prevents merchants from charging users without knowledge. It's the "two-phase commit" pattern.

Why cache the token?

bKash rate-limits token requests. Token is valid 1 hour. We cache it for 55 minutes (5-min safety buffer) so we don't request a new one before every operation.

Line-by-line explanation of the most important files.

index.js -- Entry Point ▶

// MUST be first line. Loads .env into process.env

const express = require('express');
const cors = require('cors');

// === MIDDLEWARE (runs on EVERY request, in order) ===

// === ERROR HANDLER (must be LAST) ===

// Express knows it's an error handler because of 4 params: (err, req, res, next)

app.listen(5000);
payment.js -- The Callback (most important route) ▶
router.get('/callback', async (req, res) => {
  const { paymentID, status } = req.query;
  // bKash redirects to:
  // /callback?paymentID=TR0011Vf...&status=success

  const payment = db.prepare(
    'SELECT * FROM payments WHERE bkash_payment_id = ?'
  ).get(paymentID);

  if (status === 'success') {
    // User authorized -> now EXECUTE to finalize
    const result = await bkashService.executePayment(paymentID);

    if (result.transactionStatus === 'Completed') {
      // Update DB with trxID (our proof of payment)
      db.prepare('UPDATE payments SET status=?, bkash_trx_id=? ...')
        .run('completed', result.trxID, paymentID);

      // Redirect user to React success page
      return res.redirect(`/payment-status?status=success`);
    }
  }
  // Handle cancel/failure...
});

Why res.redirect() instead of res.json()?

This endpoint is called via browser redirect (bKash sends the user's browser here), not an AJAX call. The browser is literally navigating to this URL, so we redirect to the React frontend.

Security

6 security practices implemented and why they matter.

1. Credentials never reach the frontend ▶
Wrong (insecure)

Frontend calls bKash directly with app_key and app_secret. User opens DevTools --> sees your keys --> can make fake payments.

Right (what we do)

Frontend calls OUR backend --> backend calls bKash with secrets. Secrets stay on server, user never sees them.

2. SQL Injection Prevention ▶
Vulnerable
db.prepare(`SELECT * FROM users
  WHERE name = '${userInput}'`);
// userInput = "'; DROP TABLE users; --"
// --> Deletes your entire table!
Safe (parameterized)
db.prepare(
  'SELECT * FROM users WHERE name = ?'
).get(userInput);
// ? placeholder escapes special chars
// Malicious input treated as literal string
3. CORS restricts API access ▶

We restrict which websites can call our API. Without this, any random website (evil.com) could hit our payment endpoints.

4. User PIN/OTP never touches our server ▶

User enters PIN and OTP on bKash's own page. Our server never receives, stores, or sees this data. This is PCI DSS compliance -- sensitive credentials only handled by certified processors.

5. Don't trust the frontend ▶
Wrong

Frontend says "payment successful!" --> mark order as paid. User can fake this via browser console.

Right

bKash redirects to our BACKEND. Backend calls executePayment API to VERIFY with bKash. Only if bKash confirms --> mark as paid.

6. Idempotency prevents double charges ▶

Scenario: User clicks "Pay", network slow, clicks again.

Without idempotency: Two payments created --> user charged twice!

With idempotency: Both requests use same merchantInvoiceNumber. bKash says "already exists". User charged once.

The try/catch + next(err) pattern

router.post('/create', async (req, res, next) => {
  try {
    // ... payment logic ...
    res.json(result);
  } catch (err) {
    next(err);   // Passes to errorHandler middleware
  }
});

// In middleware/errorHandler.js:
function errorHandler(err, req, res, next) {
  console.error('Error:', err.message);
  res.status(500).json({
    error: err.message || 'Internal server error',
  });
  // Never send stack traces to client (security risk)
}

Why centralized error handling?

Every route uses the same error format. Want to add Slack alerts for errors? Change ONE file. Want to log to a service? ONE file.

Production Considerations

What you'd change for a real-world production app.

1. Webhooks instead of Callbacks ▶
Callback (what we use)

bKash redirects user's browser to our URL. Problem: if user closes browser before redirect, we never know about the payment!

Webhook (production)

bKash sends a server-to-server HTTP request. No browser involved. Reliable even if user disconnects. Called IPN (Instant Payment Notification).

2. PostgreSQL instead of SQLite ▶

SQLite: one writer at a time, no network access, no replication. PostgreSQL: concurrent writes, network-accessible, replication/backup built in.

3. Redis for Token Caching ▶

Current: Token in JS variable. Server restart = token lost. Multiple servers = different tokens.

Redis: External store. Survives restarts. Shared across servers. Built-in TTL (auto-expire).

4. Payment Reconciliation ▶

Daily job: fetch all "initiated" payments, call bKash queryPayment for each, update mismatched statuses, alert team about discrepancies. Catches missed callbacks/webhooks.

5. HTTPS (SSL) ▶

bKash requires SSL for production callbacks. Sandbox allows HTTP (localhost), but production must be HTTPS.
